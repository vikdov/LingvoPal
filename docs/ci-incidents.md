# CI Incident Log

Chronological record of CI pipeline failures on `experiment/08-ci` and their fixes.
Each entry: symptom → root cause → fix → verification. Newest last.

---

## 1. Dependency CVEs + pipeline finalize
- **Commit:** `34d3863`
- **Job(s):** Backend Dependency audit; pipeline-wide
- **Symptom:** `uv audit` gate flagged known advisories; branch CI lacked full job set.
- **Root cause:** Locked deps carried known CVEs (urllib3, starlette, python-multipart, idna, mako); frontend tests had no jsdom env.
- **Fix:** Bumped affected deps; added full pipeline (lint/test/migrations/docker/bandit/audit) for both stacks; wired vitest jsdom + jest-dom; dropped `cd.yml`.

## 2. Ruff version drift
- **Commit:** `8cc8620`
- **Job(s):** Backend · Lint
- **Symptom:** Ruff format check failed in CI.
- **Root cause:** Pre-commit pinned ruff `v0.9.10` while CI/pyproject used `0.15.11` — formatter output diverged.
- **Fix:** Bumped pre-commit rev to `v0.15.11` and reformatted to match.

## 3. vitest / vite 8 type conflict
- **Commit:** `71c35b5`
- **Job(s):** Frontend · Lint & Build; Frontend · Docker build
- **Symptom:** `TS2769` in `vite.config.ts`.
- **Root cause:** vitest `3.2.4` bundled its own vite 7, conflicting with the project's vite 8 plugin types.
- **Fix:** Bumped `vitest` + `@vitest/coverage-v8` to v4 (supports vite 8).

## 4. Model drift (unmigrated schema changes)
- **Commit:** `37e21b1`
- **Job(s):** Backend · Migrations
- **Symptom:** `alembic check` reported pending autogenerate changes.
- **Root cause:** Three model changes had no migration: `items.context_audio_url` and `users.pending_email` VARCHAR→TEXT, the `uq_user_languages` unique constraint, and a column comment.
- **Fix:** Autogenerated the revision to capture the drift.

## 5. Backend base-image debian CVEs
- **Commit:** `c887b9d`
- **Job(s):** Backend · Docker build & scan (Trivy)
- **Symptom:** Trivy failed on CVEs in the debian base layer (libcap2 CVE-2026-4878, linux-libc-dev CVE-2026-43494).
- **Root cause:** gha-cached base layer carried unpatched packages.
- **Fix:** Added `apt-get upgrade -y` so base packages are patched on every build regardless of layer cache. Verified: 0 HIGH/CRITICAL on rebuilt image.

## 6. Alpine musl rolldown native binding
- **Commit:** `fe9a37a`
- **Job(s):** Frontend · Docker build
- **Symptom:** `npm ci` failed — `Cannot find module @rolldown/binding-linux-x64-musl` (vite 8 / rolldown native dep).
- **Root cause:** `node:22-alpine` (musl) lacked the glibc native binding rolldown needs.
- **Fix:** Switched builder stage to `node:22-slim` (glibc); `nginx:alpine` runtime unchanged.

## 7. .gitignore hid src/lib/utils.ts
- **Commit:** `5d4ca6b`
- **Job(s):** Frontend · Lint & Build
- **Symptom:** `TS2307 Cannot find module @/lib/utils` across 9 files in CI.
- **Root cause:** `frontend/.gitignore` had an unanchored `lib/` python-artifact rule that also matched `src/lib/`, leaving `src/lib/utils.ts` untracked. CI checked out without it.
- **Fix:** Anchored the rule to `/lib/` and committed the source file.

## 8. Frontend nginx:alpine stale base CVEs
- **Commit:** _(frontend Dockerfile `apk upgrade`)_
- **Job(s):** Frontend · Docker build & scan (Trivy)
- **Symptom:** Trivy failed with `Total: 31 (HIGH: 29, CRITICAL: 2)` — all fixable, in alpine 3.21.3 OS packages.
- **Root cause:** `nginx:1.27-alpine` base packages (libcrypto3/openssl, libxml2, libpng, libexpat, zlib, musl, nghttp2) were stale and never upgraded. Asymmetry: backend ran `apt-get upgrade`, frontend runner stage did not.
- **Fix:** Added `RUN apk upgrade --no-cache` to the nginx runner stage. Verified locally: 31 → 0 HIGH/CRITICAL, Trivy exit 0.

## 9. Trivy gate degraded by sarif format
- **Commit:** `3c44191`
- **Job(s):** Backend · Docker build & scan (Trivy)
- **Symptom:** Backend Trivy exited 1 despite the image having no HIGH/CRITICAL findings. Local scans with `--severity CRITICAL,HIGH` reported clean — the failure was not reproducible until the scanner flags matched CI exactly.
- **Root cause:** `aquasecurity/trivy-action` with `format: sarif` **ignores the `severity:` input** (log: _"Building SARIF report with all severities"_, CLI invoked with no `--severity`). So `exit-code: "1"` fired on *any* fixable finding — here 2 MEDIUM pip CVEs (`CVE-2026-3219`, `CVE-2026-6357`; pip 26.0.1 → 26.1). The `CRITICAL,HIGH` gate was never actually enforced on either Docker job.
- **Fix:**
  - Added `limit-severities-for-sarif: true` to both Trivy steps (`ci.yml:202,415`) so the SARIF — and thus exit-code — honors `severity: CRITICAL,HIGH`.
  - Bumped system pip to `>=26.1` in `backend/Dockerfile` to clear the two MEDIUM CVEs for hygiene.
  - Verified: all-severity scan of rebuilt backend image → exit 0.

## Recurring patterns / notes
- **Trivy + `format: sarif` discards `severity`** — always pair with `limit-severities-for-sarif: true` if the gate should be severity-scoped. (Incident 9)
- **gha layer cache hides base-image CVEs** — `apt-get upgrade` / `apk upgrade` in the image keeps packages patched regardless of cache. (Incidents 5, 8)
- **Tooling version drift** between pre-commit and CI/pyproject causes lint/format failures. (Incidents 2, 3)
- **Transient infra** seen on one run (cache service `400`, "services aren't available") = GitHub outage, not config.
